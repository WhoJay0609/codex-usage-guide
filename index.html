Desktop and CLI

先认识 Codex Desktop 和 CLI

Codex Desktop 更像一个可视化工程工作台，适合多线程、worktree、浏览器检查和 review pane；Codex CLI 更像终端里的自动化执行器，适合脚本化、approval modes、sandbox、MCP、subagents 和 codex exec。

Codex Desktop 适合什么

  • 多线程工作同时开多个任务线程，让调查、实现、review 分开推进。
  • 可视化审查适合看 diff、浏览器页面、截图、review pane 和运行结果。
  • 仓库协作适合本地项目、worktree、GitHub Pages、前端页面和需要人工判断的迭代。
  • 长任务跟踪适合持续观察进展、补充约束、批准命令和最终验收。

Desktop 第一条任务

把项目打开后，不要直接说“帮我优化”。先让它读 AGENTS.md、README 和相关文件，再要求小范围修改和验证。

Desktop 的优势

上下文、视觉反馈、人工批准和审查体验更强，适合交互式开发。

Codex CLI 适合什么

  • 终端执行在 shell 里直接用 codex 交互，保留命令行工作习惯。
  • 脚本化用 codex exec 做非交互任务、批处理、CI 辅助或结构化输出。
  • 权限控制通过 sandbox、approval policy、profiles 和 config 控制读写、网络和命令批准。
  • 工具连接适合 MCP、subagents、web search、自动化和团队内部工作流。

CLI 第一条任务

codex "read AGENTS.md and explain how to run tests"

CLI 自动化入口

codex exec "review this diff and output risks"

Permissions and evidence

权限安全要先讲清楚

Codex 能读写代码、运行命令、访问网络或云端环境。每次授权前都要问：它为什么需要这个权限、会改哪里、失败后如何恢复、最后用什么证据证明完成。

Read-only

适合调查、审查、解释错误、整理计划。只读任务不应改文件，也不应偷偷生成 artifact。

Auto / workspace-write

适合明确范围内的普通实现。要求 Codex 保留 unrelated dirty worktree，并先说明会改哪些路径。

Full access / network

只在确实需要下载依赖、访问 GitHub API、验证线上页面或使用外部服务时开放，并要求说明理由。

Cloud / GitHub

适合 PR review、云端任务和 CI 自动化。要把 secret、权限、环境差异和回滚策略写清楚。

Customization

把长期规则放进项目，而不是每次重说

公开文档把 AGENTS.md、Skills、MCP、Subagents、Memories 都放在 customization 体系下。它们解决的是不同层次的问题：项目规则、可复用技能、外部工具、并行角色和长期记忆。

AGENTS.md

写项目规则：搜索方式、编辑边界、测试命令、敏感路径、最终报告格式。根目录写通用规则，子目录写局部规则。

Skills

写可复用流程：比如工程 grilling、学术 pipeline、部署检查。skill 应该有触发场景、输入、输出和验证方式。

MCP / Plugins

连接外部系统：设计工具、数据库、GitHub、文档库、搜索服务。使用前明确权限和数据边界。

推荐 AGENTS.md 骨架

## Repository Expectations
- Preserve unrelated dirty worktree changes.
- Use rg for search and apply_patch for manual edits.

## Verification
- Run the narrowest relevant test first.
- Report changed files, commands, failures, and residual risk.

## Boundaries
- Do not edit secrets or generated files unless requested.

什么时候沉淀成规则

如果你连续两次纠正 Codex 同一类问题，就不要第三次靠聊天提醒。把它写入 AGENTS.md、Skill、README、ADR 或测试。

Goal mode

Goal 是执行合同，不是标题

当任务跨阶段、跨仓库、需要 artifact、需要团队协作或需要清晰 closeout 时，用 Goal。Goal 写得越具体，Codex 越容易在长上下文里保持方向。

Objective

一句话写清最终可观察成果。

Scope

写 include 和 exclude，尤其是不能改什么。

Acceptance

写用户如何验收，不只写测试命令。

Stop Conditions

写什么时候必须停下来报告 blocked。

Objective:
<one concrete outcome sentence>

Scope:
- Include: <paths/modules/tasks>
- Exclude: <explicitly out of scope>

Success Criteria:
- <observable result>

Acceptance Criteria:
- <what the user can check>

Validation:
- <commands, screenshots, logs, links>

Stop Conditions:
- <when to stop and report blockers>

Subagents

并行不是目的，隔离上下文才是目的

Subagent 适合独立调查、只读审查、并行测试、互不重叠的实现切片。主线程仍要负责整合、判断和最终验收。

请用并行 subagents 审查当前分支。
spawn one agent per point, wait for all, then summarize.

Agent 1: security risks，只读审查，不改文件。
Agent 2: missing tests，只读审查，不改文件。
Agent 3: maintainability，只读审查，不改文件。

每个 agent 返回 findings、risk level、recommended fix、tests/evidence。

Practical flows

把技能串成可执行流程

工程和学术场景都不要停留在“介绍 skill”。真正有用的是：什么时候进入、输入什么、产出什么、用什么证据判断可以进入下一阶段。

工程 flow：idea 到可合并变更

  • 路由ask-matt 判断是 grill、PRD、issues、implement 还是 review。
  • 打磨grill-with-docs 读 AGENTS.md、README、CONTEXT、ADR，并把稳定结论写回文档。
  • 拆解to-prd 写需求边界，to-issues 拆成可验证小任务。
  • 执行一次只做一个 issue，跑最窄验证，最后报告 diff、测试和风险。
[$grill-with-docs]
仓库：/path/to/repo
目标：把功能 X 做成可发布版本。
请先读 AGENTS.md、README、相关源码，再追问关键边界。
稳定术语写入 CONTEXT.md；重大结构决策写入 docs/adr/。

学术 flow：idea 到可审查证据

  • Brief先写 RESEARCH_BRIEF.md：方向、数据、baseline、预算、目标会议、不能 claim 什么。
  • 强审research-grill-me 检查问题重要性、closest work、实验可证伪性和 claim 边界。
  • Pipelineresearch-pipeline 产出 IDEA_REPORT.md、实验、AUTO_REVIEW.md、NARRATIVE_REPORT.md。
  • Gate首次使用建议 AUTO_PROCEED: false，确认 idea 和预算后再 full experiment。
[$research-pipeline]
方向：长上下文 LLM 推理中的 KV cache pruning。
RESEARCH_BRIEF.md: 请先创建并让我确认。
AUTO_PROCEED: false
HUMAN_CHECKPOINT: true
REVIEWER_DIFFICULTY: hard
AUTO_WRITE: false
预算：最多 20 GPU-hours。

自动化 flow：从交互到脚本

当同一类任务反复出现，再考虑 codex exec、GitHub Action、JSONL 输出或内部脚本。自动化前先把权限、输入、输出和失败策略固定下来。

codex exec "review this PR for missing tests and output JSONL"

自动化前检查：
- 是否需要网络或 secret
- 是否能限制写入路径
- 是否有结构化输出
- 失败时是否停止而不是继续破坏状态

Local scenario

goal-entry：本机 Goal 路由器

goal-entry 是本机维护的扩展，不是 OpenAI 官方内置能力。它希望把复杂请求先分类，再决定是否创建 Goal、绑定 active goal、只做计划、只做报告，或进入 repo-bound 执行链路。

Resolver

resolve_goal_entry.py 产出 request_mode、goal_entry_tier、goal_action、dispatch level 和 readiness。

Preflight

goal-preflight 拦截 hard blockers：目标超过 4,000 字符、缺 repo/path、上下文要求停止、readiness 未通过。

Child contracts

goal-objective、goal-context、goal-dispatch、goal-trace、goal-close 分别负责目标、上下文、团队、证据和收口。

希望达成的功能

  • 不误执行讨论类请求不自动改文件，执行类请求先过 gate。
  • 不空泛Goal 保留成功标准、验收标准、验证和停止条件。
  • 可收口最终用 evidence closeout，而不是一句“完成”。

可复制提示词

[$goal-entry]
执行：把 codex-usage-guide 改成中文 GitHub Pages 指南。
要求：先做 preflight；如果 ready=false，报告 blockers。
验收：页面已发布、关键章节可跳转、最终报告 commit 和验证。

Design council

团队讨论后的指南设计原则

这次重设计把指南定位为“Codex 101 中文实战手册”，不是官方文档的逐字翻译，也不是本机 skills 的展示页。读者先得到稳定工作模型，再进入工程、学术和本机扩展。

先解决入口选择

App、CLI、IDE、Cloud、GitHub integration 的差异要前置，否则读者不知道该在哪里开始。

权限安全单独成章

sandbox、approval、network、API key、Cloud 环境不能混在实现教程里一笔带过。

Prompt 必须可复制

所有流程都要落到“我可以直接粘贴给 Codex”的输入模板，而不是停留在原则。

本机扩展必须降噪

goal-entry 要清楚标成本机扩展，不应被视觉上包装成 OpenAI 官方能力。

高级视觉不能牺牲文档可读性

页面可以有 editorial hero、dense bento、horizontal accordion、marquee 和 scroll motion，但代码块、锚点、资料链接、移动端扫描体验必须优先。

最重要的内容缺口

  • 入口矩阵读者要知道什么任务用什么入口。
  • 安全解释读者要知道何时批准命令和网络。
  • 端到端案例从 issue 到 diff、测试、PR review、fix。
