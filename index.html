Getting started

零基础 10 分钟上手

回到顶部

第一次使用 Codex 时，不要先追求复杂自动化。先跑通一个最小闭环：选择项目、写清任务、让 Codex 读上下文、执行、验证、看证据。

打开项目

在 Codex app 中选择明确项目文件夹，不要从无关的大目录开始。

写四段式请求

目标、范围、约束、验证。哪怕只写四行，也比“帮我优化”可靠。

让 Codex 先读上下文

要求读取 AGENTS.md、README.md、相关源码或 SKILL.md。

看证据再接受

结束时检查 diff、测试、构建、日志、网页、截图或其它可观察产物。

Prompt contract

最小请求模板

回到顶部 
在这个项目里做一件事：<目标>。
范围：只改 <路径/模块>，不要动 <排除范围>。
约束：<语言、格式、API、字段、权限、风格等硬约束>。
验证：完成后请运行 <命令/检查>，并告诉我改了哪些文件、验证结果和剩余风险。
判断完成的标准：不要只看 Codex 是否说“完成”，要看 diff、测试、构建、日志、网页、截图或其它可观察证据。

Goal mode

Goal 模式怎么写

回到顶部

Goal 不是标题，而是一份执行合同。它适合长任务、跨阶段任务、需要 artifact、验证和 closeout 的任务。

必须包含

Objective、Scope、Out of Scope、Success Criteria、Acceptance Criteria。

必须约束

不能改什么、权限边界、输出语言、格式、公开 API、停止条件。

必须验证

测试命令、构建命令、截图、日志、人工验收、剩余风险。

Objective:
<one concrete outcome sentence>

Scope:
- Include: <paths/modules/tasks>
- Exclude: <explicitly out of scope>

Validation:
- <commands to run>
- <manual checks or artifact checks>

Stop Conditions:
- <when to stop and report blocked>

Subagent mode

Subagent 怎么用

回到顶部

Subagent 的价值是隔离上下文，把独立探索、测试、日志分析和审查移出主线程。它不是随便并行，必须有明确边界和输出合同。

请用并行 subagents 审查当前分支。
spawn one agent per point, wait for all of them, then summarize.

Agent 1: security risks，只读审查，不改文件。
Agent 2: missing tests，只读审查，不改文件。
Agent 3: maintainability，只读审查，不改文件。

每个 agent 返回 findings、risk level、recommended fix、tests/evidence。

Project rules

项目里的 AGENTS.md

回到顶部

AGENTS.md 是项目说明书。根文件放共识，子目录文件放局部规则。越靠近目标路径的规则优先级越高。

仓库期望

搜索方式、编辑方式、保留 unrelated dirty worktree、禁止 destructive 操作。

构建与测试

列出最窄验证命令：单测、构建、lint、链接检查、GitHub Pages 发布检查。

完成证据

要求最终说明改了哪些文件、跑了什么、失败原因、剩余风险。

# AGENTS.md

## Repository Expectations
- Preserve unrelated dirty worktree changes.
- Use apply_patch for manual edits.
- Prefer rg / rg --files for search.

## Verification
Before claiming completion, report:
- files changed
- commands run
- tests passed or why they could not run
- remaining risks

Scenario · Goal router

场景：用 goal-entry 启动复杂任务

回到顶部

goal-entry 是本机维护的 Goal 工作流入口，不是 OpenAI 官方内置 skill。它的定位是 router-only：先把用户请求分类，再决定是否创建 Goal、绑定已有 Goal、只做计划、只做报告，或进入 repo-bound 的执行链路。这样做的核心价值是避免长任务一上来就误执行、误建 Goal、误派 subagent，或者跳过验收和 closeout。

公开仓库

codex-goal-entry 已单独发布，包含 SKILL.md、resolver、架构说明和 standalone validator。

使用时机

跨仓库发布、长流程执行、多阶段研究、需要 readiness gate、需要 subagent 边界、需要 artifact 和 closeout 的任务。

不该使用

简单问答、单文件小改、纯 review、一次性命令、用户明确说“只讨论，不执行”的场景。

架构：一个薄入口，多层内部合同

goal-entry 自己不承载全部策略。它只负责稳定入口和路由，真正的执行约束分散到 child skills 和脚本里，这样入口不会越来越重，也方便单独验证每一层。

Public entry

goal-entry/SKILL.md 是用户看到的唯一入口。旧的 harness-agent-for-goal、harness-agent 等仍保留兼容，但不再鼓励直接作为新入口。

Resolver

scripts/resolve_goal_entry.py 先产出 request_mode、goal_entry_tier、goal_action、dispatch level 和 readiness 状态。

Preflight gate

goal-preflight 在 execute_goal 或 active_goal_bind 前运行。未 ready 时停止，不创建 Goal，不编辑文件，不派 subagent。

Objective contract

goal-objective 负责把目标压成可执行合同：成功标准、验收标准、停止条件都要保留，且 compact objective 不能超过 4,000 字符。

Context contract

goal-context 负责 repo-bound 上下文，包括 AGENTS.md 指令链、路径索引、可编辑范围和必要 artifact 目录。

Dispatch & closeout

goal-dispatch、goal-team、goal-backend、goal-trace、goal-close 负责团队、运行记录、证据和完成收口。

用户请求
  -> goal-entry/SKILL.md
  -> resolve_goal_entry.py
     -> report_only        只回答，不建 Goal
     -> plan_only          只规划，读 goal-plan
     -> execute_goal       先 preflight，再 objective/context/dispatch/backend
     -> active_goal_bind   绑定已有 Goal，先 preflight，再继续执行
  -> goal-preflight
     -> ready=false        停止并报告 blockers
     -> ready=true         才允许创建/绑定 Goal 和执行
  -> goal-trace + goal-close
     -> 用证据关闭，而不是只说“完成”

实际执行 flow

Step 1: 分类请求

先用 resolver 判断请求是 report_only、plan_only、execute_goal 还是 active_goal_bind。分类结果优先于手写判断。

Step 2: plan-only 只规划

如果用户只是要方案、边界、问题确认或架构讨论，读取 goal-plan，不要创建 Goal，也不要开始改文件。

Step 3: 执行前 preflight

执行类请求必须先跑 goal-preflight。常见 hard blockers：目标超过 4,000 字符、缺少 repo/path、上下文要求停止、readiness 未通过。

Step 4: 写完整 Goal

Goal 里必须有 objective、scope、out of scope、success criteria、acceptance criteria、validation 和 stop conditions。

Step 5: 装载项目上下文

repo-bound 任务读取 AGENTS.md、生成或使用路径索引、确认敏感文件和编辑边界，再决定是否需要 artifact run。

Step 6: 组建团队

只有当任务能拆出独立工作时才派 subagent。subagent 不能调用 get_goal、create_goal、update_goal，也不能拿到 goal-* 协议 skill。

Step 7: 执行并留痕

执行过程要产生 artifact、命令、日志、diff、测试或发布证据。复杂运行使用 backend/trace 记录，而不是只靠聊天总结。

Step 8: closeout

结束前读取 goal-trace 和 goal-close，确认验收、验证、剩余风险、子进程清理和最终状态。

希望达成的功能

防止误执行

用户只是要讨论时，不自动进入实现；用户要执行时，也先过 readiness gate。

防止空泛 Goal

Goal 不只是标题，要包含成功标准、验收标准、验证命令、停止条件和范围边界。

让 subagent 可控

主线程保留 Goal 权限和最终判断，subagent 只做明确范围内的调查、实现或验证。

让 repo 上下文稳定

通过 AGENTS.md、路径索引和 artifact 约定，让 Codex 不靠猜测理解项目结构。

让完成可验证

closeout 必须带证据：改动、测试、构建、日志、线上链接、失败项和剩余风险。

让 skill 可发布

通过 standalone 仓库、validator 和架构文档，让其它用户能理解、安装和二次维护。

可复制提示词

[$goal-entry]
执行：创建 goal-entry 的公开仓库，推送到 GitHub；
然后更新 codex-usage-guide，在介绍 goal-entry 时链接新仓库。

验收：
- 新仓库公开可访问；
- resolver quick_validate 通过；
- codex-usage-guide 的 GitHub Pages 已更新；
- 最终报告链接、提交 hash 和验证结果。
[$goal-entry]
只做计划，不执行：
我想把一个研究项目从 idea 推到 workshop submission。
请先判断是否需要 Goal、需要哪些 artifact、哪些阶段必须人工确认。
不要创建 Goal，不要改文件，先给我边界问题和推荐 flow。
[$goal-entry]

使用原则：先分类，再执行。不要让 subagent 创建、更新或关闭 Goal；不要跳过 readiness gate。

Engineering workflow

工程场景：从想法到可合并变更

回到顶部

工程 skills 的重点不是“调用了哪个 skill”，而是把模糊需求推进成可审查、可实现、可验证的变更。下面是一条实际可操作的 flow：先路由，再追问，再沉淀上下文，再拆 PRD 和 issues，最后小步实现与验证。

入口判断

ask-matt 先判断当前请求应该走 idea 打磨、项目文档、PRD、issue 拆分、实现还是 review。

边界打磨

grill-me 适合无仓库 idea；grill-with-docs 适合有仓库并需要维护 CONTEXT.md、ADR 和项目约束。

落到执行

to-prd 写需求边界，to-issues 拆小任务，implement 或普通 Codex 执行单个 issue。

Step 0: 写清任务合同

用“目标、范围、约束、验证”四段式，不要只写“优化一下”。如果任务超过半小时，先写 Goal 或让 goal-entry 判断是否进入 Goal。

Step 1: 先路由

不确定该用哪个 skill 时，先让 ask-matt 判断。输出必须包含：推荐 skill、为什么、下一步要读哪些文件、是否需要先追问。

Step 2: 用 grill-with-docs 压边界

让 Codex 读取 AGENTS.md、README.md、CONTEXT.md、相关源码和已有 ADR。每次只追问关键问题，稳定结论写回文档。

Step 3: 写 PRD

用 to-prd 产出目标用户、核心 workflow、非目标、验收标准、风险、发布方式和回滚策略。PRD 不要混入实现细节。

Step 4: 拆 issues

用 to-issues 把 PRD 拆成 1-3 小时能完成的任务。每个 issue 写清影响文件、测试命令、完成定义和不该改的边界。

Step 5: 单 issue 执行

一次只让 Codex 实现一个 issue。要求先读代码，再 patch，再跑最窄验证，最后报告 diff、测试、失败项和剩余风险。

Step 6: 审查与发布

进入 review stance：先列 bug、回归、测试缺口，再总结。网页项目要检查本地页面、链接、GitHub Pages 状态和线上关键文本。

Step 7: 沉淀经验

如果这次工作暴露了稳定规则，把它写入 AGENTS.md、CONTEXT.md 或 ADR，而不是只留在聊天记录里。

可复制实例：把一个仓库改造成 GitHub Pages 中文指南

[$ask-matt]
我有一个已有仓库 /path/to/codex-usage-guide。
目标：把它改成可以发布到 GitHub Pages 的中文 Codex 使用指南。
请判断应该先用 grill-with-docs、to-prd、to-issues 还是直接实现。
输出：推荐路径、需要读取的文件、必须追问的问题、下一条提示词。
[$grill-with-docs]
仓库：/path/to/codex-usage-guide
目标：做成中文 GitHub Pages 指南，不再以 PDF 为主。
请先读 AGENTS.md、README.md、CONTEXT.md、现有 index.html。
然后追问我最多 3 个会影响结构的关键问题。
稳定术语写入 CONTEXT.md；重大结构选择写入 docs/adr/。
[$to-prd]
基于刚才 grill-with-docs 的结论，写一个 PRD。
必须包含：
- 目标读者：零基础用户、进阶用户、维护者
- 核心页面结构：侧边栏、卡片、锚点跳转、中文内容
- 非目标：不生成 PDF，不做复杂前端框架
- 验收：GitHub Pages 可访问，关键章节可跳转，示例提示词可复制
- 风险：本机自写 skills 不能伪装成官方能力
[$to-issues]
把 PRD 拆成 issues。
每个 issue 必须包含：
- 背景和目标
- 修改范围
- 验收标准
- 最窄验证命令
- 不允许修改的内容

优先拆成：
1. Web 页面结构
2. Goal/Subagent/AGENTS.md 内容
3. 工程 flow 内容
4. 学术 flow 内容
5. 发布与链接验证
执行 issue 3：工程 flow 内容。
范围：只改 index.html 的 #engineering section。
要求：不要写成 skill 简介，要写成可操作 flow；加入 3 段以上可复制提示词。
验证：运行 git diff --check，并用 rg 检查 ask-matt、grill-with-docs、to-prd、to-issues 是否都出现。
工程验收清单：最终必须看到 PRD 或等价需求、issues 或等价任务拆分、实际 diff、验证命令、发布状态、剩余风险。只有“我已经更新了”不算完成。

Academic workflow

学术场景：从研究想法到可审查证据

回到顶部

学术 skills 的重点是防止“先写论文再找证据”。一个实际可操作的 flow 应该先限定研究问题和资源，再做文献与新颖性检查，再进入实验，最后用自动 review 卡住过度 claim。

academic-research-suite

总入口，适合不知道该走综述、审稿、实验计划、写作还是完整 pipeline 的情况。

ask-research

研究路由器。先判断该用 research-grill-me、research-pipeline、文献检索、实验计划还是论文润色。

research-grill-me

强门槛问答，适合把一个 idea 压到“是否值得做、怎么证明、不能 claim 什么”。

research-pipeline

端到端流水线：idea discovery、实现、实验、auto-review-loop、写作 handoff。

实际可操作 flow

Step 0: 建立研究 brief

先写 RESEARCH_BRIEF.md：研究方向、目标会议、数据集、baseline、算力预算、不能使用的数据、预期贡献类型。

Step 1: 先路由

用 ask-research 判断当前是 idea 太粗、文献不足、实验设计不足，还是已经可以进入 research-pipeline。

Step 2: 用 research-grill-me 卡门槛

依次检查问题是否重要、closest work 是否接近、方法差异是否实质、实验是否能排除替代解释、claim 是否越界。

Step 3: 进入 Stage 1 idea discovery

让 research-pipeline 生成 idea-stage/IDEA_REPORT.md，包含 top ideas、closest work、pilot 信号、novelty 风险和推荐理由。

Step 4: 人工 gate

首次使用建议 AUTO_PROCEED: false。确认 idea、预算、停止条件后，再允许进入实现和 full experiment。

Step 5: 实现实验

把 pilot 扩成可复现实验：固定数据、baseline、metric、seed、日志目录、表格格式和失败记录。

Step 6: 跑实验

小批量直接 run-experiment；长任务进入 experiment-queue。每轮都记录命令、commit、配置、GPU-hours 和输出路径。

Step 7: 自动 review

用 auto-review-loop 生成 review-stage/AUTO_REVIEW.md。如果 review 认为证据不足，先补实验，不要提前写 strong claim。

Step 8: 写作 handoff

通过门槛后生成 NARRATIVE_REPORT.md，再进入写作。写作只引用已经落盘、可追溯的图表和日志。

可复制实例 1：从零开始找一个可做 idea

[$ask-research]
我想做一个 AI/ML 研究项目，但 idea 还很粗。
方向：长上下文 LLM 推理中的 KV cache pruning。
目标：8 周内做出可投稿 workshop 的实证结果。
资源：2 张 24GB GPU，每周最多 20 GPU-hours。
请先判断应该走 research-grill-me、research-pipeline 还是文献综述。
输出：推荐路径、必须补充的信息、第一步提示词。
[$research-grill-me]
研究方向：长上下文 LLM 推理中的 KV cache pruning。
请用中文强审这个 idea。
重点追问：
1. closest work 是什么，差异是否真实；
2. 只做 engineering trick 是否不够；
3. 哪个实验能证明方法不是偶然；
4. 什么 claim 现在绝对不能写；
5. 如果只给 20 GPU-hours，最小 pilot 怎么做。
[$research-pipeline]
方向：长上下文 LLM 推理中的 KV cache pruning。
RESEARCH_BRIEF.md: 请先创建并让我确认。
AUTO_PROCEED: false
HUMAN_CHECKPOINT: true
REVIEWER_DIFFICULTY: hard
AUTO_WRITE: false
预算：最多 20 GPU-hours。
停止条件：如果 novelty 风险高、baseline 无法复现、或 pilot 没有正信号，停止并报告。

请先生成 idea-stage/IDEA_REPORT.md，列出 top ideas、closest work、
pilot 设计、预期表格和推荐理由。不要在我确认前进入 full experiment。

已有材料：

[$research-pipeline]
执行已确认的实验计划 docs/experiment_plan.md。
AUTO_PROCEED: true
HUMAN_CHECKPOINT: false
AUTO_WRITE: false
REVIEWER_DIFFICULTY: hard
预算：今晚最多 12 GPU-hours。
并发：最多 2 个任务。
失败策略：任一 baseline 失败两次就停止队列，不要继续烧 GPU。

输出要求：
- 每个实验的命令、配置、commit、日志路径；
- 成功/失败表；
- 明早可读的 review-stage/AUTO_REVIEW.md；
- 不写论文正文。
学术验收清单：进入写作前至少要有 RESEARCH_BRIEF.md、IDEA_REPORT.md、可复现实验命令、原始日志、汇总表、AUTO_REVIEW.md 和明确 claim 边界。没有这些证据时，不要让 Codex 直接写“可投稿论文”。
