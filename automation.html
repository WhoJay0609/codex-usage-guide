Advanced task

自动化

自动化不是把 Codex 放开让它自己跑，而是把稳定任务封装成受控输入、受控权限、结构化输出和明确失败策略。适合自动化的任务，必须先能人工跑通。

When

什么时候值得自动化

同一类任务反复出现，输入可以参数化，输出可以验证，失败能安全停止，这时才考虑 Codex Desktop Automations。先人工跑通，再让后台任务按计划重复执行。

适合 Automations

批量 review、生成结构化报告、检查文档漂移、分析日志、定期跑本地站点检查。

适合后台 worktree

对 Git 仓库中的重复任务使用独立后台 worktree，避免直接污染未完成的本地工作区。

适合本地脚本

稳定输入输出的仓库任务，例如生成索引、检查链接、归档实验结果。

自动化前置条件

  • 输入稳定同类任务重复出现，参数能结构化。
  • 输出可验JSONL、测试结果、PR comment 或文件产物。
  • 失败会停权限、网络、测试失败时必须停止并报告。

不该自动化的任务

  • 需求未定还需要人判断产品方向或论文 claim。
  • 权限过宽需要生产 secret、外部写入或不可回滚操作。
  • 验证缺失没有测试、没有审查点、没有输出结构。

Patterns

三种实用自动化模式

自动化的关键不是命令本身，而是输入、权限、输出和失败策略。

只读审查

Automation prompt:
Review this diff for missing tests.
Output JSONL with fields:
severity, file, finding, evidence, recommendation.
Do not edit files.

失败日志解释

Automation prompt:
Read the latest failure log and explain root cause.
Return:
- failing command
- likely cause
- minimal fix
- whether local reproduction is needed.

文档漂移检查

Automation prompt:
Compare README, AGENTS.md, and project scripts.
Report stale commands, missing setup steps, and risky assumptions.
Do not modify files.

Safety

自动化安全阀

自动化最危险的地方，是它把一次错误变成重复错误。每个自动化任务都应写明权限和停止条件。

最小权限

能只读就只读；能限定目录就限定目录；需要网络时说明访问目标。

结构化输出

优先 JSONL、表格或固定字段，避免自动化结果无法被脚本和人复核。

人工 checkpoint

涉及写文件、发布、PR comment、外部系统写入时保留人工确认点。

失败即停止

权限失败、测试失败、上下文缺失、网络不可达时不要继续猜测执行。

自动化任务模板：
Input:
- repo/path:
- target files:
- task type:

Permission:
- read:
- write:
- network:
- external systems:

Output:
- format:

Stop:
- missing context

- ambiguous requirement
- unsafe permission request

Real example

真实实例：自动检查网页链接和关键标题

这份手册每次修改后，都可以把“本地链接检查 + 公开页面关键标题抽样”沉淀为自动化。它适合自动化，因为输入稳定、输出可验、失败能安全停止。

输入

坏输入是“每天帮我看一下”；修复后写清频率、项目、输出字段和停止条件。

先读

README、AGENTS.md、验证脚本、最近失败日志和自动化权限设置。

应动文件

自动化 prompt、检查脚本或报告路径；不直接改生产 secret 或外部系统。

验证

先在普通 Desktop 线程手动跑通，再安排 Automations。

最终回答

列出 findings、证据路径、是否需要人工处理和下一次运行条件。

失败停止

权限不足、输入缺失、网络不可达、验证脚本失败时不要继续猜测。
