    进阶任务

    Scheduled tasks（定时任务）

    定时任务不是把 Codex 放开让它自己跑，而是把稳定任务封装成受控输入、受控权限、结构化输出和明确失败策略。适合定时运行的任务，必须先能在普通任务中人工跑通。

    何时使用

    什么时候值得自动化#

    同一类任务反复出现，输入可以参数化，输出可以验证，失败能安全停止，这时才考虑 Scheduled tasks（定时任务）。先人工跑通，再让后台任务按计划重复执行。

    适合定时任务#

    批量审查、生成结构化报告、检查文档漂移、分析日志、定期跑本地站点检查。

    适合后台 worktree#

    对 Git 仓库中的重复任务使用独立后台 worktree，避免直接污染未完成的本地工作区。

    适合本地脚本#

    稳定输入输出的仓库任务，例如生成索引、检查链接、归档实验结果。

    自动化前置条件#

    • 输入稳定同类任务重复出现，参数能结构化。
    • 输出可验JSONL、测试结果、PR comment 或文件产物。
    • 失败会停权限、网络、测试失败时必须停止并报告。

    不该自动化的任务#

    • 需求未定还需要人判断产品方向或论文 claim。
    • 权限过宽需要生产 secret、外部写入或不可回滚操作。
    • 验证缺失没有测试、没有审查点、没有输出结构。

    配置

    在 Desktop 或 Web 中创建定时任务#

    打开 Scheduled，选择新建任务；结果与历次运行也在 Scheduled 视图查看。项目任务只有在机器开机且 ChatGPT desktop app 正在运行时才能执行。

    逐项配置#

    1. 描述要完成的工作，并设置 schedule / cadence（执行时间或频率）。
    2. 选择继续当前任务，还是创建独立任务。
    3. 项目任务选择 Local 或新 worktree；不确定时优先新 worktree 隔离改动。
    4. 选择模型与推理强度，并核对默认 sandbox（沙箱）权限。
    5. 保存前先把同一提示词作为普通任务跑通；上线后人工审查前几次运行。

    配置模板#

    任务：检查文档命令是否与仓库脚本一致
Schedule：每周一 09:00（Asia/Shanghai）
上下文：继续当前任务 / 独立任务
环境：Local / 新 worktree
模型与推理：按任务复杂度选择
权限：默认 sandbox；禁止外部写入
输出：文件、证据、风险、下一步
停止条件：命令失败、需要 secret 或范围不明时停止

    模式

    三种实用自动化模式#

    自动化的关键不是命令本身，而是输入、权限、输出和失败策略。

    只读审查#

    定时任务提示词：
Review this diff for missing tests.
Output JSONL with fields:
severity, file, finding, evidence, recommendation.
Do not edit files.

    失败日志解释#

    定时任务提示词：
Read the latest failure log and explain root cause.
Return:
- failing command
- likely cause
- minimal fix
- whether local reproduction is needed.

    文档漂移检查#

    定时任务提示词：
Compare README, AGENTS.md, and project scripts.
Report stale commands, missing setup steps, and risky assumptions.
Do not modify files.

    安全

    自动化安全阀#

    定时任务最危险的地方，是它把一次错误变成重复错误。每个任务都应写明权限和停止条件。

    最小权限#

    能只读就只读；能限定目录就限定目录；需要网络时说明访问目标。

    结构化输出#

    优先 JSONL、表格或固定字段，避免自动化结果无法被脚本和人复核。

    人工 checkpoint#

    涉及写文件、发布、PR comment、外部系统写入时保留人工确认点。

    失败即停止#

    权限失败、测试失败、上下文缺失、网络不可达时不要继续猜测执行。

    无人值守禁区#

    不要在无人值守的 Scheduled tasks 中运行 /lfg，也不要调用任何会自动 commit、push、打开 PR 或反复修复 CI 的 skill。仓库应启用 protected branch，任务使用最小权限；任何写入、提交与发布结果都必须由人审查，merge 只由人执行。

    定时任务模板：
Input:
- repo/path:
- target files:
- task type:

Permission:
- read:
- write:
- network:
- external systems:

Output:
- format:
- required fields:
- artifact path:

Stop:
- missing context
- command failure
- ambiguous requirement
- unsafe permission request

    真实实例

    真实实例：自动检查网页链接和关键标题#

    这份手册每次修改后，都可以把“本地链接检查 + 公开页面关键标题抽样”沉淀为自动化。这个任务适合自动化：输入稳定、输出可验，失败时也能安全停止。

    输入#

    坏输入是“每天帮我看一下”；修复后写清频率、项目、输出字段和停止条件。

    先读#

    README、AGENTS.md、验证脚本、最近失败日志和自动化权限设置。

    应动文件#

    自动化 prompt、检查脚本或报告路径；不直接改生产 secret 或外部系统。

    验证#

    先在普通 Desktop 任务手动跑通，再安排 Scheduled tasks；发布后在 Scheduled 视图审查前几次运行。

    最终回答#

    列出 findings、证据路径、是否需要人工处理和下一次运行条件。

    失败停止#

    权限不足、输入缺失、网络不可达、验证脚本失败时不要继续猜测。
