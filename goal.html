    Concept

    Goal

    Goal 是长任务的执行合同，不是标题。它把目标、范围、验收、验证和停止条件写清楚。

    What

    它是什么#

    Goal 适合跨阶段、跨仓库、需要 artifact、需要 subagent 或需要 closeout 的任务。好的 Goal 会保留成功标准、验收标准、验证命令和停止条件。

    When

    什么时候用 / 不用#

    • 用 Goal跨多个页面或仓库、需要发布、需要实验、需要 subagent、需要最终 closeout。
    • 不用 Goal简单问答、单文件小改、一次性解释、纯只读 review。
    • 写法要点Objective 写结果，Scope 写边界，Acceptance 写用户如何验收，Validation 写命令和证据。

    Examples

    可复制示例#

    Objective:
把指南拆成多页面 GitHub Pages 文档站。

Scope:
- Include: HTML pages, assets/site.css, assets/site.js
- Exclude: PDF 和 LaTeX

Acceptance:
- 每个页面可访问，导航可跳转
- GitHub Pages built

Stop Conditions:
- 官方资料无法访问
- 共享资源导致页面不可读

    Design

    一个好 Goal 必须包含什么#

    Goal 的价值是让长任务不会在上下文里漂移。它不是标题，也不是一句愿望，而是能被执行、能被验收、能在失败时停止的合同。

    Objective#

    一句话写最终可观察结果。不要写“优化系统”，要写“把搜索页迁移到新 API 并保持旧入口兼容”。

    Scope#

    列 include 和 exclude。include 告诉 Codex 能动哪里；exclude 防止它顺手改架构、格式化无关文件或改变公开接口。

    Acceptance#

    写用户如何验收。测试通过只是工程证据，用户验收还可能包括页面能访问、文案正确、PR check 变绿。

    Validation#

    写命令、截图、线上状态、日志和人工检查点。无法验证时要要求 Codex 保留风险说明。

    Artifacts#

    写最终应产生什么：commit、PR、报告、网页、实验表、图、README 或 ADR。

    Stop Conditions#

    写什么时候必须停：权限不足、官方资料不可访问、测试环境缺失、需求冲突、证据不足。

    Examples

    三种常见 Goal 模板#

    网页发布 Goal:
Objective: 把中文 Codex 指南扩成可发布的 GitHub Pages 手册。
Scope: include HTML/CSS/JS/docs; exclude PDF/LaTeX。
Acceptance: 首页、日常页、Goal 页、资源页公开可访问。
Validation: git diff --check、本地静态服务、GitHub Pages built。
Stop: 官方资料无法访问或页面资源 404。

工程修复 Goal:
Objective: 修复 checkout 优惠券逻辑导致的失败测试。
Scope: include checkout/coupon/tests; exclude payment provider API。
Acceptance: 指定测试通过，旧支付流程不回归。
Validation: npm test -- checkout；必要时跑 coupon tests。
Stop: 失败来自环境或需求冲突。

研究自动化 Goal:
Objective: 完成 KV cache pruning idea 的文献、新颖性和小规模实验审查。
Scope: include brief、closest work、实验脚本、AUTO_REVIEW；exclude 论文全文写作。
Acceptance: 有 research brief、实验日志、负结果记录和 claim 边界。
Validation: 实验命令、结果表、引用链接。
Stop: baseline 不可复现或 GPU 预算不足。

    Real example

    真实实例：历史复合案例#

    历史复合案例完整案例的 Goal 摘录

    请求：把一个多页面指南改成证据驱动的团队工作手册。角色与边界：用户批准 Product Contract 和发布动作；主线程维护统一计划与 Git 集成；执行者只改分配页面。共享样式、公共导航、私人证据和既有 dirty 修改均不在写域。

    Objective:
让读者能从任务入口走到可复核的 done 或 blocked receipt，并理解 Worktree × subagent 的协作边界。

Scope:
- Include: 既有任务、工程、Goal、上下文、subagent 与第三方仓库页面；静态校验
- Exclude: 新页面、全局导航重构、共享 CSS/JS、PDF、私人过程记录

Acceptance:
- 五阶段循环和团队交付路径可互相跳转
- 历史、复合与演示案例可见且语义一致
- Worktree 与 subagent 的权限、恢复和停止条件完整

Validation:
- 语义负向测试和全站静态检查
- desktop / narrow browser sampling
- diff、独立 review、PR/CI 与授权后的公开页面抽样

Artifacts:
- 更新后的 HTML、案例证据索引、校验器测试、PR 和 task receipt

Stop Conditions:
- 来源或脱敏证据不足、写域重叠、权限/依赖缺失、review 或 CI 未通过、merge 未授权

    验证与重要恢复#

    预检发现主 checkout 有无关修改后，没有清理或覆盖，而是把写入切到隔离范围；每个执行切片返回 changed files 与检查证据，主线程再运行全站验证。

    结果与回执#

    证据全部 accepted 时签发 completed receipt；权限、来源、依赖或最终验证仍缺失时签发 blocked receipt，并列出已验事实、解锁条件和下一步。完整来源分类与交付链见 团队协作历史复合案例。

    停止条件：任何越界写入、私人标识泄露、未授权外部动作或无法复现的成功声明都会停止发布。该摘录由多个已核验片段组成，不声称来自一次连续执行。
