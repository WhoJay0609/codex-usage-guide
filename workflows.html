Tasks

任务路径总览

本指南默认你主要使用 Codex Desktop。先按桌面版任务闭环组织工作，再决定是否需要 Goal、subagent、skills 或自动化。任务页的重点是可执行流程，不是术语解释。

Route

先判断任务类型

好的 Desktop 请求不是“帮我做一下”，而是交代你当前打开的页面或仓库、目标、范围、验收、验证方式和不能触碰的边界。越是长任务，越应该把这些内容显式写出来。

开 Desktop 线程

一个线程对应一个目标。把当前 URL、仓库路径、截图/评论、目标读者和不可修改范围放在开头。

给上下文

说明关键文件、已有失败日志、相关 issue/PR，并要求 Codex 先读 AGENTS.md 再行动。

定验收

提前写出“完成意味着什么”：页面能访问、测试通过、diff 可审、结论有证据、没有改动禁区。

收证据

最终回答应包含改了什么、审批了什么、跑了什么、没跑什么、风险在哪里、远程链接或 commit 是什么。

Codex Desktop

先学桌面版线程、浏览器反馈、审批、diff review 和发布验证。

日常小任务

读代码、解释错误、修 bug、跑测试、审查 diff，通常不需要 Goal。

工程 flow

复杂功能先 grill，再 PRD，再 issues，再实现和 review。

学术 flow

从 research brief 到文献、新颖性、实验、审稿和写作边界。

自动化

把稳定重复任务放进 codex exec、GitHub integration 或 CI。

goal-entry

Prompt skeleton

通用任务模板

背景：
我正在 Codex Desktop 里处理 [项目/论文/网页]。
当前页面/仓库是：[URL 或 /path/to/repo]。
当前问题是：[具体现象]。

目标：
请完成 [可验收结果]。

范围：
- 可以改：[文件/目录/页面]
- 不要改：[禁区/外部接口/已有用户改动]

要求：
- 先读 AGENTS.md、README 和相关源码
- 改动前说明你准备改哪里
- 需要联网、启动服务或推送前先说明理由
- 完成后运行最窄验证

验收：
- [测试/构建/页面访问/截图/commit/push]
- 最终报告列出改动、验证、风险和未完成项

Real example

真实实例：从一句反馈选择任务路径

用户在 Codex Desktop 的浏览器里说“每个内容都加一个真实的实例”。这不是简单问答，也不是纯规划；它会修改多个页面、需要验证、需要发布，所以任务路径应当是：Desktop 主线程 + Goal 风格执行任务 + GitHub Pages 发布验证。

路径选择：
1. goal-entry 路由和 preflight
2. 读取 AGENTS.md、页面和相关 skill
3. 给每个主要页面添加真实实例
4. 本地静态验证
5. git commit / push
6. GitHub Pages built 后抽样公开 HTML
