Tasks

任务路径总览

先按你要完成的工作选择路线，再决定是否需要 Goal、subagent、skills 或自动化。任务页的重点是可执行流程，不是术语解释。

Route

先判断任务类型

好的 Codex 请求不是“帮我做一下”，而是交代背景、目标、范围、验收、验证方式和不能触碰的边界。越是长任务，越应该把这些内容显式写出来。

选入口

需要浏览器、截图、页面发布和多线程时用 Desktop；需要终端批处理、CI、MCP 或非交互时用 CLI。

给上下文

说明仓库路径、关键文件、已有失败日志、相关 issue/PR、目标读者和不可修改范围。

定验收

提前写出“完成意味着什么”：页面能访问、测试通过、diff 可审、结论有证据、没有改动禁区。

收证据

最终回答应包含改了什么、跑了什么、没跑什么、风险在哪里、远程链接或 commit 是什么。

Desktop / CLI

先解决入口选择：交互式协作、终端执行、IDE 内编辑或远程任务。

日常小任务

读代码、解释错误、修 bug、跑测试、审查 diff，通常不需要 Goal。

工程 flow

复杂功能先 grill，再 PRD，再 issues，再实现和 review。

学术 flow

从 research brief 到文献、新颖性、实验、审稿和写作边界。

自动化

把稳定重复任务放进 codex exec、GitHub integration 或 CI。

goal-entry

Prompt skeleton

通用任务模板

背景：
我在 /path/to/repo 做 [项目/论文/网页]，当前问题是 [具体现象]。

目标：
请完成 [可验收结果]。

范围：
- 可以改：[文件/目录/页面]
- 不要改：[禁区/外部接口/已有用户改动]

要求：
- 先读 AGENTS.md、README 和相关源码
- 改动前说明你准备改哪里
- 完成后运行最窄验证

验收：
- [测试/构建/页面访问/截图/commit/push]
- 最终报告列出改动、验证、风险和未完成项
