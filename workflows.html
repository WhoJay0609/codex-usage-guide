任务路径总览

本指南默认你主要使用 Codex Desktop。先按桌面版任务闭环组织工作，再决定是否需要 Goal、subagent、skills 或自动化。任务页的重点是可执行流程，不是术语解释。

Route

先选工作面，再写任务

好的 Desktop 请求不是“帮我做一下”，而是交代你当前打开的页面或仓库、目标、范围、验收、验证方式和不能触碰的边界。越是长任务，越应该先选入口。

页面/视觉Desktop + in-app browser不要只说“优化一下”URL、截图/评论、页面抽样
代码/测试Desktop + integrated terminal不要跳过失败日志失败命令、最窄测试、diff
多文件长任务Goal + 明确验收不要把多个目标塞进一轮objective、计划、验证、closeout
重复任务Automations + worktree不要人工循环催办Triage inbox、run 记录、停止条件

Prompt skeleton

统一案例模板

背景：
我正在 Codex Desktop 里处理 [项目/论文/网页]。
当前页面/仓库是：[URL 或 /path/to/repo]。
当前问题是：[具体现象]。

目标：
请完成 [可验收结果]。

范围：
- 可以改：[文件/目录/页面]
- 不要改：[禁区/外部接口/已有用户改动]

要求：
- 先读 AGENTS.md、README 和相关源码
- 改动前说明你准备改哪里
- 需要联网、启动服务或推送前先说明理由
- 完成后运行最窄验证

验收：
- [测试/构建/页面访问/截图/commit/push]
- 最终报告列出改动、验证、风险和未完成项

Counterexamples

坏 prompt 怎么修

反例的价值是暴露缺失边界。每次修 prompt，都补齐范围、验收、验证和停止条件。

“帮我优化项目”

改成：说明目标页面/目录、目标读者、允许改的文件、验收命令和不碰的边界。

“直接推送”

改成：先报告 diff 和验证结果，再说明推送目标和公开页面抽样方式。

“开 subagent 并行做”

改成：列出每个写域、等待点、输出合同、集成负责人和冲突处理方式。

Real example

真实实例：从一句反馈选择任务路径

用户在 Codex Desktop 的浏览器里说“每个内容都加一个真实的实例”。这不是简单问答，也不是纯规划；它会修改多个页面、需要验证、需要发布，所以任务路径应当是：Desktop 主线程 + Goal 风格执行任务 + GitHub Pages 发布验证。

路径选择：
1. 选择 Desktop 线程和 Goal 风格验收
2. 读取 AGENTS.md、页面和相关资料
3. 给每个主要页面添加真实实例
4. 本地静态验证
5. git commit / push
6. GitHub Pages built 后抽样公开 HTML
