Local adaptation

goal-entry

goal-entry 是本机维护的 Goal 路由器，不是 OpenAI 官方内置能力。它展示了如何把“用户一句话”变成可执行、可验证、可停止的 Goal 工作流。

Positioning

它解决什么问题

普通聊天任务很容易混淆：用户是在问问题、要计划、要执行、继续已有目标，还是要组织团队？goal-entry 的目标是先分类，再选择正确工作模式。

不误执行

讨论类请求不自动改文件；缺少 repo/path、验收标准或上下文时先报告 blockers。

不空泛

Goal 必须保留 objective、scope、acceptance、validation、artifacts、stop conditions。

可收口

最终用 evidence closeout 报告产物、验证、风险和未完成项，而不是一句“完成”。

Architecture

架构和处理流程

这套设计把 Goal 入口拆成 resolver、preflight、child contracts 和 closeout，而不是让一个大 prompt 同时做所有判断。

Resolver

resolve_goal_entry.py 判断 request_mode、goal_action、dispatch level 和 readiness。

Preflight

goal-preflight 拦截目标过长、缺 repo/path、上下文停止、readiness 未通过。

Child contracts

goal-objective、goal-context、goal-dispatch、goal-trace、goal-close 各自负责一层协议。

四步处理流程

1. 识别请求区分问答、计划、执行、继续 active goal。
2. 做 preflight检查目标长度、repo/path、上下文和 readiness。
3. 分发团队必要时进入 objective、context、dispatch、trace。
4. 证据收口用 closeout 报告产物、验证、风险和阻塞。

Examples

什么时候使用 goal-entry

它适合长任务入口，不适合简单问答。使用时要给目标、范围、验收和停止条件。

网页指南任务

[$goal-entry]
执行：把 codex-usage-guide 扩成中文 Codex 实战手册。

范围：
- 扩充 HTML 页面和共享样式
- 不维护 PDF

验收：
- daily workflow 内容足够丰富
- 进阶章节有详细流程和模板
- GitHub Pages built

停止条件：
- 官方资料无法访问
- 页面资源 404
- 需要用户确认定位

工程修复任务

[$goal-entry]
执行：修复 checkout 优惠券测试失败。

要求：
- 先识别是否 ready
- 如果缺失败日志，先报告 blocker
- 如果 ready，创建 Goal 并按最小 diff 实现

验收：
- npm test -- checkout 通过
- 最终 closeout 包含改动、验证、风险

Failure modes

失败模式和防护

本机扩展最重要的是降低误执行概率，而不是让 Codex 更激进。

把讨论当执行

防护：resolver 必须区分 question、plan、execute；讨论类请求不创建 Goal。

Goal 太长或太空

防护：preflight 检查目标长度和必要字段；缺 acceptance 或 stop conditions 时先追问。

团队分发过度

防护：只有独立子任务才 dispatch；主线程保留决策和最终 closeout。

公开边界

goal-entry 是 local adaptation。公开指南应把它作为“本机扩展示例”，不要包装成 OpenAI 官方功能。

迁移建议

迁移到其它机器时，先确认 skill 文件、resolver 脚本、子 skill、验证脚本和仓库路径是否存在，再使用同名 prompt。
