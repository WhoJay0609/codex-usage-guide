Why change the organizing principle

Grounding Context

当前指南已经有 19 个 HTML 页面，首页按读者状态和任务类型分流，任务页普遍包含可复制 prompt、验证、停止条件和真实实例；make check 也能验证 HTML、站内链接、锚点、导航、图片和关键文本。这些基础意味着下一阶段不应再以“新增页面”作为成功标准。

仓库证据显示，真正的缺口集中在五处：读者不知道何时从普通 Desktop 任务升级；读完后没有可观察的掌握标准；失败时常只有“停止”而缺少恢复路径；产品与第三方事实没有统一保鲜机制；维护者也不知道读者是否真的完成了任务。新增检查还发现，指南虽提到 Worktree 和 subagent，却没有解释两者在多个代理同时修改代码时承担的不同隔离责任。

外部研究同样支持这个判断：任务优先导航优于概念目录；最小化教学应让读者立刻完成真实任务，并把错误识别与恢复纳入教程；成熟文档系统会测试链接与内容契约；服务型指南应以完成率和首次阻塞点衡量效果。

核心判断：这份指南现在缺的不是更多知识，而是一个把“入口、练习、证据、恢复、反馈”串起来的学习合同。

Two combined directions worth discussing

Ranked Ideas

用户选择合并原有方向，因此第一条吸收任务护照、阶梯练习、恢复卡、事实供应链和完成基准的优点；第二条补齐 Worktree 与多个写入型 subagent 的隔离方法。两条都可以单独进入 ce-brainstorm。

1

Idea 1. 证据驱动任务闭环：把五个方向整理成一条读者主线

用“选择、练习、验收/恢复、事实治理、反馈”统一任务页，而不是继续扩张页面。

Axis
任务发现与进阶
Confidence
94%
Complexity
Medium
Description

只让任务页承担完整闭环：先用任务长度、写入范围、外部状态、验证证据和重复度选择入口；再通过 2-3 个同题阶梯练习修正 prompt；完成时检查实际证据，失败时进入共享恢复卡；涉及产品或第三方事实时读取可核验来源；最后用隐私保守的任务回执记录是否完成和首次阻塞。概念页和参考页继续保持轻量，不强制复制整套闭环。

Basis

direct: 指南已经分别拥有任务分流、反例 prompt、closeout、停止条件和来源分层，但这些能力散落在不同页面，尚未形成统一读者合同。external: TaskNavigator、Minimalist Instruction、Diátaxis、GitLab 文档测试和 GOV.UK completion guidance 分别支持任务优先、错误恢复、渐进披露、事实验证和任务完成度量。

Rationale

它吸收原五个方向的优点，同时避免把四套新机制强制塞进每一页。主线任务页负责“做成”，概念页负责“解释”，维护层负责“事实仍然有效”。

Downsides
如果一次性把五个环节完整铺到所有页面，会重新制造内容膨胀。应只在核心任务页呈现完整闭环，并允许分阶段验证每个环节是否有效。
2

Idea 2. Worktree × subagent：两层隔离的并行写入方法

Worktree 隔离任务 checkout，subagent 隔离职责；两者不能互相替代。

Axis
安全执行、恢复与隔离
Confidence
93%
Complexity
Medium
Description

Worktree 的作用是让多个独立 Codex 任务各自在单独的 Git checkout 中工作，不干扰 Local 或其它任务；subagent 的作用是把同一任务拆成有边界的子工作。不要假设同一任务中的每个 subagent 自动拥有独立 worktree：若它们在同一任务中并行写代码，仍要按“同一工作区可能冲突”设计，明确互不重叠的文件所有权、等待点和主线程集成责任。

Basis

external: OpenAI 官方 Worktrees 文档说明 Worktree 为同一项目中的独立任务提供不同 checkout，并支持从 Worktree 创建分支或 Hand off 到 Local；官方 Subagents 文档同时提醒，并行写入型工作流可能产生冲突和协调开销。direct: 当前指南已经要求 subagent 的写域互不重叠，但尚未把这一规则与 Worktree 的任务级隔离连起来。

Rationale

这能纠正常见误解：“用了 Worktree，多个代理就不会冲突。”真正安全的模型是两层：不同任务之间用 Worktree 隔离 checkout；同一任务内部用文件所有权隔离 subagent 写域，主线程负责合并、测试和最终 Git 操作。

How to use
  1. 确认项目是 Git 仓库；在 Codex Desktop 新任务的 composer 下选择 Worktree。
  2. 选择起始分支。Codex 默认基于该分支创建 detached HEAD 的 managed worktree。
  3. 在 prompt 中列出每个 subagent 的目标、可写文件、禁止路径、输出合同和等待点；共享核心文件、schema 或 API contract 时改为串行。
  4. 要求所有 subagent 完成后，由主线程检查 diff、处理冲突并运行统一测试；subagent 不独立 commit、push 或发布。
  5. 验证完成后，在 Worktree 中使用 Create branch here 提交并开 PR，或使用 Hand off 把任务和代码带回 Local。
  6. 同一 branch 不能同时 checkout 在两个 worktree。需要在 Local 使用该 branch 时，优先 Hand off，不要强行重复 checkout。
请在当前 Worktree 中并行使用 3 个 subagents：
Agent A：只修改 frontend/，完成界面实现。
Agent B：只修改 tests/，补行为测试。
Agent C：只读审查接口和冲突风险，不改文件。

约束：
- 写域不得重叠，不修改共享 schema 和 API contract
- subagents 不创建 branch/worktree，不 commit、不 push
- 等待全部返回后，由主线程整合 diff、运行测试并报告冲突
Downsides
每个 Worktree 都有完整文件、副本依赖和缓存，会占用磁盘；多个任务如果修改同一核心设计，最终仍会发生语义冲突。重叠写域应串行处理，不能用更多 Worktree 掩盖架构协调问题。
