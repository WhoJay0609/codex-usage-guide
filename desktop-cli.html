Codex Desktop

桌面版怎么用

这份手册只把 Codex Desktop 作为主入口。Desktop 的优势是把对话、仓库、内置终端、浏览器反馈、权限审批、Git diff 和发布验证放在同一个工作面里，适合交互式地完成真实工程任务。

When

什么时候优先用 Desktop

如果任务需要你边看结果边调整，或者需要 Codex 解释它正在做什么，Desktop 应该是默认入口。重复任务也先在 Desktop 跑通，再沉淀为 Automations。

页面和视觉反馈

网页、PDF、截图、排版、交互状态、GitHub Pages 发布效果，适合放在 Desktop 里迭代。

本地仓库协作

让 Codex 读 AGENTS.md、源码、测试、diff，再通过 Desktop 内置终端运行验证命令。

需要你审批的操作

安装依赖、联网查询、启动本地服务、推送 Git，都可以在 Desktop 里看到原因后再批准。

沉淀为 Automations 的场景

批量审查、周期检查、文档漂移、链接检查和固定报告，先在 Desktop 线程跑通，再放进 Automations 后台执行。

不要混用得太乱

同一个任务建议有一个主线程负责判断。不要在多个 Desktop 线程里同时改同一批文件，除非写范围严格隔离。

Operations

Desktop 里的六个常用操作

零基础用户不需要先背概念。按下面六步走，基本能覆盖网页、文档、小型代码修改和发布验证。

1. 打开任务线程

一个线程只放一个目标。开头写清仓库路径、页面 URL、当前问题、验收标准和禁区。

2. 让 Codex 读上下文

要求先读 AGENTS.md、README、相关页面或源码。复杂任务再让它跑 rg 找入口。

3. 用浏览器反馈定位问题

如果你在 in-app browser 里看到页面不对，把当前 URL、区域截图或评论直接发给 Codex。

4. 审批有副作用动作

遇到联网、启动服务、推送、安装依赖等请求，先看 Codex 给出的理由，再决定是否批准。

5. 看 diff 和验证结果

让 Codex 报告改了哪些文件、跑了哪些命令、哪些页面或测试已经通过。

6. 发布后抽样确认

如果是 GitHub Pages 或 PR，要求 Codex 等状态完成后抽样公开 URL，而不是只说“已 push”。

Prompt patterns

Desktop-first prompt 模板

Desktop 任务的 prompt 应该包含“你正在看的界面”和“希望 Codex 如何验证”。这比只说“改一下网页”稳定得多。

网页反馈

我在 Codex Desktop 的浏览器里打开：
https://example.github.io/project/page.html

问题：
- 当前区域内容太少
- 需要补充 Desktop 操作步骤

请先读 AGENTS.md 和相关 HTML。
只改网页内容，不改视觉风格。
完成后本地抽样访问，再推送并验证公开 URL。

代码修复

我在 Desktop 里处理这个仓库：
/path/to/repo

现象：
- 测试 npm test -- checkout 失败

要求：
- 先读 AGENTS.md 和失败日志
- 改动前说明根因
- 只改 checkout 相关文件
- 完成后运行最窄测试并报告 diff

需要联网或推送

如果需要联网、启动本地服务或 git push，
请先说明：
- 为什么需要这个权限
- 会访问或写入哪里
- 成功后如何验证

不要在没有说明的情况下执行外部写操作。

线程收口

最终请按这个格式回答：
改动：
- [文件] [变化]

验证：
- [命令/URL] [结果]

交付：
- commit: [hash]
- 公开链接: [URL]

风险：
- [未验证项或人工确认项]

Real example

真实实例：本页这次怎么改

用户在 Desktop 浏览器里打开 workflows.html，要求“主要面向 Codex Desktop，把桌面版操作加进去”。这是典型 Desktop 任务：用户给出页面上下文，Codex 读仓库规则和页面，修改 HTML，运行检查，推送后验证公开站点。

输入

当前 URL、用户反馈、项目规则和目标是 Desktop-first。

执行

重写本页，补充线程、浏览器反馈、审批、diff、发布验证等操作。

验证

git diff --check、站内链接检查、本地或公开 HTML 关键标题抽样。

交付

提交、推送，并确认 GitHub Pages 状态和线上页面内容。
